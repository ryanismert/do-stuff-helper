# W9: Backlog & Replanning Process

## Objective

Establish a backlog capture convention and a replan skill that periodically processes accumulated ideas into roadmap and brief updates.

## Done When

- Workers add discovered tasks directly to the current waypoint's task list during implementation (convention documented in CLAUDE.md)
- A `docs/backlog.md` file exists as the standard place to capture ideas, scope changes, and discovered work
- A `replan` skill reads the backlog, categorizes items, and proposes changes to the brief and roadmap
- Categorization: **design update** (modifies the brief + may reorder/change waypoints) or **tactical waypoint** (new discrete waypoint added to the roadmap)
- The skill presents proposed changes to the user for confirmation before applying
- On confirmation: updates the brief, updates the roadmap JSON (add/modify/reorder waypoints), and clears processed items from the backlog
- Changes are committed with a clear summary

## Design

### Backlog File

`docs/backlog.md` — append-only, freeform:

```markdown
## <date> — <source>

<description>
```

Source can be a worker ("during w18 implementation"), the user, or another skill. No strict schema — the replan skill interprets intent from natural language.

### In-Flight Task Convention

Documented in CLAUDE.md / waypoint-implement skill: when a worker discovers additional work needed for the current waypoint, it creates tasks on the task list for that waypoint and does them. No backlog entry needed — it's just part of the implementation.

The backlog is for things that are **out of scope** for the current waypoint.

### Replan Skill Checklist

1. Read `docs/backlog.md`, `docs/roadmap-<slug>.json`, `docs/brief-<slug>.md`
2. For each backlog item, categorize as:
   - **Design update** — changes understanding of the activity, its goals, or approach. Results in brief updates and potentially waypoint reordering/modification.
   - **Tactical waypoint** — new discrete work item. Gets added to the roadmap as a new waypoint with dependencies.
3. Present the categorized list with proposed changes to the user
4. On confirmation, apply: update the brief, update the roadmap JSON, clear processed items
5. Commit with changelog summary

### Cross-Skill Invocation

- **Invoked by user** — "let's replan", "review the backlog", "process the backlog"
- **Fed by workers** — waypoint-implement workers append to backlog when they discover out-of-scope work
- **Could be scheduled** — n8n weekly trigger (future, when inbox/dashboard exist)
