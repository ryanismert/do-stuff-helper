# W11: Inbox

## Objective

Give workers a way to ask questions asynchronously and give the user a way to see and answer those questions from any device, decoupled from active Claude sessions.

## Done When

- Workers can send questions that persist beyond the current session
- The dashboard shows pending questions with context (what the worker was doing, which task/waypoint)
- The user can reply to questions from the dashboard UI (including mobile)
- The implement skill picks up answered questions and resumes work
- Questions and answers are stored per-activity in `docs/inbox.json`
- An n8n workflow detects answered questions and triggers implement to resume automatically
- The `waiting` waypoint status prevents duplicate implement sessions

## Design

### Current State

The implement skill already handles worker questions (Step 3), but it's synchronous — the user must be in the same Claude session. If the session ends, questions are trapped in task metadata with no visibility.

### Waypoint Status: `waiting`

Add a new status to the waypoint lifecycle:

| Status | Meaning |
|--------|---------|
| `pending` | Not started |
| `implementing` | A session is actively working |
| `waiting` | Blocked on user — questions or human tasks outstanding |
| `done` | Completed |
| `obsolete` | No longer relevant |

The implement skill sets `waiting` at its deliberate exit point — when it has exhausted all dispatchable tasks and only blocked/question tasks remain. This is reliable because it's a controlled state transition, not a crash-cleanup situation.

### End-to-End Flow

```
Implement starts     → sets status to "implementing"
Dispatches workers, processes results...
Worker flags QUESTION: → implement writes to inbox.json, parks the task
Implement continues with other unblocked tasks...
All remaining tasks blocked on user → sets status to "waiting", exits

[user answers questions on dashboard from phone]

n8n polls: status == "waiting" AND answered entries in inbox.json?
  Yes → trigger Claude webhook to resume implement
  No  → do nothing (either still "implementing" or no answers yet)

New session starts   → sets "implementing", reads answered entries, continues
All tasks done       → sets "done"
```

### Storage: `docs/inbox.json`

Per-activity file, array of messages:

```json
[
  {
    "id": "q-1",
    "task_id": "14",
    "waypoint": "w10",
    "subject": "[w10] Build dashboard UI",
    "question": "Should the dark theme use pure black or GitHub-style dark?",
    "context": "Building the dashboard HTML. Both approaches work but have different visual feels.",
    "created": "2026-02-25T12:00:00Z",
    "status": "pending",
    "answer": null,
    "answered_at": null
  }
]
```

**Statuses:** `pending` (waiting for reply), `answered` (user replied via dashboard), `resolved` (implement skill picked up the answer and re-dispatched the task)

### Dashboard Integration

**API endpoints** (added to the existing dashboard server):

- `GET /api/inbox` — returns all pending/answered questions across activities
- `POST /api/inbox/:activity/:id/reply` — accepts `{ "answer": "..." }`, updates inbox.json with answer and sets status to `answered`

**Note:** The dashboard container currently mounts activities as read-only (`:ro`). This needs to change to read-write for the reply endpoint to write to inbox.json.

**UI additions:**

- **Inbox section** at the top of the dashboard (above activity cards) showing pending questions across all activities
- Each question shows: waypoint badge, task subject, question text, context
- Reply form: text input + submit button per question
- Answered questions collapse/fade but remain visible until resolved

### n8n Resume Workflow

An n8n workflow polls for answered questions and triggers implement to resume:

1. **Schedule trigger**: runs every 5 minutes
2. **Read inbox files**: scan `~/exoselfai/activities/*/docs/inbox.json` for entries with `status: "answered"`
3. **Check waypoint status**: read the corresponding roadmap JSON — only trigger if the waypoint is in `waiting` status (not `implementing`, which means a session is already running)
4. **Trigger webhook**: POST to `localhost:3001/webhook/execute` with payload specifying the activity and waypoint to resume
5. **Idempotent**: if the implement session picks up the answers before the webhook fires, the new session finds nothing to do and exits cleanly

### Implement Skill Changes

**Step 1b (Set Status):** Set status to `implementing` as before.

**Step 3 (Check for Worker Questions):** In addition to checking `metadata.question`, read `docs/inbox.json` for entries with `status: "answered"`. For each:
- Append the answer to the task description as "Additional Context"
- Update the inbox entry to `status: "resolved"`
- Clear `metadata.question` on the task
- Task is now ready for re-dispatch

**Step 5 (worker QUESTION: handling):** When a worker flags a question:
- Write the question to `docs/inbox.json` (append to array, auto-increment ID)
- Still set `metadata.question` for in-session visibility
- Commit the inbox.json change so it's visible to the dashboard

**Step 7 (Loop exit — waiting condition):** When the loop exits because all remaining tasks are blocked (questions pending or human tasks):
- Set waypoint status to `waiting` in the roadmap JSON
- Report to user what's waiting and how to answer (dashboard URL)

### Relationship to Blockers

Blockers (`docs/blockers.json`) and inbox questions serve different purposes:
- **Blockers**: "This task requires human action" (make a phone call, create an account) — needs action + resolution, no reply
- **Questions**: "I need clarification to proceed" — needs a text reply

Both show on the dashboard but in different sections. No consolidation.

### Edge Cases

- **Session crashes while `implementing`**: Status stays stuck. n8n won't trigger (waiting for `waiting`). User notices on dashboard ("active" but nothing happening) and re-invokes manually. Could add staleness heuristic later.
- **Multiple questions answered at once**: Handled naturally — Step 3 processes all answered entries in a single loop iteration.
- **User answers in a Claude session instead of dashboard**: Still works — implement's Step 3 checks both metadata.question (in-session) and inbox.json (dashboard). The in-session path clears the inbox entry too.
- **No inbox.json exists**: Skills create it with `[]` on first write. Dashboard returns empty array.

### What's NOT in Scope

- Push notifications when a question arrives (that's w17)
- Rich formatting in questions or answers (plain text)
- Threaded conversations (one question, one answer per entry)
- Proactive worker messages that don't need a reply (future)
