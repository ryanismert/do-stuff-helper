# W11: Inbox

## Objective

Give workers a unified way to ask questions and flag human tasks asynchronously, and give the user a way to see, reply, resolve, or discuss these items from any device — decoupled from active Claude sessions.

## Done When

- Workers can send questions and human-task blockers that persist beyond the current session
- All user-facing items (questions and blockers) are stored in a single `docs/inbox.json` per activity, replacing the separate `docs/blockers.json`
- The dashboard shows pending inbox items with context (what the worker was doing, which task/waypoint)
- The user can resolve items via three paths: **reply** (text answer), **resolve** (mark done + optional note), or **discuss** (launch a conversation session with Claude)
- The implement skill picks up answered/resolved items and resumes work
- An n8n workflow detects resolved items and triggers implement to resume automatically
- The `waiting` waypoint status prevents duplicate implement sessions
- The discussion channel is pluggable — initial implementation uses a fallback (manual session), with a research spike to evaluate Claude remote control and Telegram as richer options

## Design

### Current State

The implement skill handles two kinds of user-facing items:
- **Worker questions** (Step 3) — synchronous, user must be in the same Claude session. If the session ends, questions are trapped in task metadata.
- **Human-task blockers** (Step 2) — written to `docs/blockers.json`, visible on the dashboard but with no resolution mechanism beyond manual status updates.

Both are "things waiting on the user" and should share a single storage format, dashboard UX, and resume flow.

### Waypoint Status: `waiting`

Add a new status to the waypoint lifecycle:

| Status | Meaning |
|--------|---------|
| `pending` | Not started |
| `implementing` | A session is actively working |
| `waiting` | Blocked on user — questions or human tasks outstanding |
| `done` | Completed |
| `obsolete` | No longer relevant |

The implement skill sets `waiting` at its deliberate exit point — when it has exhausted all dispatchable tasks and only blocked/question tasks remain. This is reliable because it's a controlled state transition, not a crash-cleanup situation.

### End-to-End Flow

```
Implement starts     → sets status to "implementing"
Dispatches workers, processes results...
Worker flags QUESTION: → implement writes to inbox.json (type: question), parks the task
Worker hits human task → implement writes to inbox.json (type: blocker), parks the task
Implement continues with other unblocked tasks...
All remaining tasks blocked on user → sets status to "waiting", exits

[user resolves items on dashboard from phone — reply, mark done, or discuss]

n8n polls: status == "waiting" AND resolved/answered entries in inbox.json?
  Yes → trigger Claude webhook to resume implement
  No  → do nothing (either still "implementing" or no answers yet)

New session starts   → sets "implementing", reads answered entries, continues
All tasks done       → sets "done"
```

### Storage: `docs/inbox.json`

Per-activity file, array of items. Replaces the previous `docs/blockers.json` format.

**Question example:**

```json
{
  "id": "q-1",
  "type": "question",
  "task_id": "14",
  "waypoint": "w10",
  "subject": "[w10] Build dashboard UI",
  "body": "Should the dark theme use pure black or GitHub-style dark?",
  "context": "Building the dashboard HTML. Both approaches work but have different visual feels.",
  "created": "2026-02-25T12:00:00Z",
  "status": "pending",
  "resolution": null,
  "resolved_at": null
}
```

**Blocker example:**

```json
{
  "id": "q-2",
  "type": "blocker",
  "task_id": "7",
  "waypoint": "w6",
  "subject": "[w6] Set up DNS records",
  "body": "CNAME record needed: app.example.com → hosting-provider.com",
  "context": "Deployment is ready but the domain isn't pointed yet.",
  "created": "2026-02-25T13:00:00Z",
  "status": "pending",
  "resolution": null,
  "resolved_at": null
}
```

**Fields:**
- `type` — `"question"` (needs a text answer) or `"blocker"` (needs human action + confirmation)
- `body` — the question text or description of what the user needs to do
- `resolution` — the user's reply text (for questions) or a note about what they did (for blockers, optional)
- `resolved_at` — timestamp when the user responded

**Statuses:** `pending` (waiting on user), `answered` (user responded via dashboard or discussion), `resolved` (implement skill picked up the response and re-dispatched the task)

### Dashboard Integration

**API endpoints** (added to the existing dashboard server):

- `GET /api/inbox` — returns all pending/answered items across activities
- `POST /api/inbox/:activity/:id/reply` — accepts `{ "resolution": "..." }`, updates inbox.json and sets status to `answered`
- `POST /api/inbox/:activity/:id/resolve` — accepts `{ "resolution": "..." }` (optional note), updates inbox.json and sets status to `answered`
- `POST /api/inbox/:activity/:id/discuss` — triggers a discussion session (see Discussion Channel below)

**Note:** The dashboard container currently mounts activities as read-only (`:ro`). This needs to change to read-write for the reply/resolve endpoints to write to inbox.json.

**UI additions:**

- **Inbox section** at the top of the dashboard (above activity cards) showing pending items across all activities
- Each item shows: type badge (question/blocker), waypoint badge, task subject, body text, context
- **Three resolution actions per item:**
  - **Reply** (questions) — text input + submit button. User types an answer.
  - **Resolve** (blockers) — "Mark done" button with optional note field. User confirms they did the thing.
  - **Discuss** (either type) — "Discuss with Claude" button. Launches a conversation session to work through the item interactively. See Discussion Channel below.
- Answered/resolved items collapse/fade but remain visible until the implement skill marks them `resolved`

### Discussion Channel

Some inbox items can't be resolved with a text box — they need a back-and-forth conversation with Claude. The "Discuss" action launches a Claude session pre-loaded with the inbox item's context.

**Architecture:** The discussion channel is a pluggable concern. The dashboard triggers a session and provides the user a way to connect to it. The mechanism can be swapped without changing the inbox model or the implement skill.

**Candidate mechanisms (to be evaluated via research spike):**

1. **Claude Code remote control** — Dashboard triggers a webhook that starts a Claude session focused on the inbox item. User connects via Claude's remote control feature from any device. *Unknowns: requires app (not web?), one connection per session, session lifecycle management.*

2. **Telegram bot** — Dashboard provides a deep link to a Telegram conversation with a Claude-backed bot, pre-loaded with context. Works via Telegram web client or native apps. *Pulls w17 (Chat Integration) forward partially. Requires bot infrastructure + relay service.*

3. **Manual session (fallback)** — Dashboard displays the inbox item ID and a command to paste into a terminal (e.g., `claude` then "work on inbox item q-3"). A small skill or command loads the item context and works through it interactively. *No infrastructure needed, but requires terminal access.*

**Initial implementation:** Use the manual session fallback (option 3). The dashboard shows the item ID and a suggested command. A `/inbox` command or skill loads the item context. This is functional from day one with zero infrastructure.

**Research spike:** Evaluate options 1 and 2 as a task within w11. If remote control works well, wire it up. If not, evaluate Telegram. The dashboard UI just needs to swap what the "Discuss" button does.

### n8n Resume Workflow

An n8n workflow polls for answered/resolved inbox items and triggers implement to resume:

1. **Schedule trigger**: runs every 5 minutes
2. **Read inbox files**: scan `~/exoselfai/activities/*/docs/inbox.json` for entries with `status: "answered"`
3. **Check waypoint status**: read the corresponding roadmap JSON — only trigger if the waypoint is in `waiting` status (not `implementing`, which means a session is already running)
4. **Trigger webhook**: POST to `localhost:3001/webhook/execute` with payload specifying the activity and waypoint to resume
5. **Idempotent**: if the implement session picks up the answers before the webhook fires, the new session finds nothing to do and exits cleanly

This works identically for both questions and blockers — the n8n workflow doesn't care about the `type`, only that `status` changed to `answered`.

### Implement Skill Changes

**Step 1b (Set Status):** Set status to `implementing` as before.

**Step 2 (Human Tasks):** When surfacing human tasks, write them to `docs/inbox.json` as `type: "blocker"` instead of `docs/blockers.json`. The blockers.json file is retired.

**Step 3 (Check for Answered Items):** In addition to checking `metadata.question`, read `docs/inbox.json` for entries with `status: "answered"`. For each:
- If `type: "question"`: append the `resolution` text to the task description as "Additional Context"
- If `type: "blocker"`: the task's human prerequisite is satisfied; task is unblocked
- Update the inbox entry to `status: "resolved"`
- Clear `metadata.question` on the task (if applicable)
- Task is now ready for re-dispatch

**Step 5 (worker QUESTION: handling):** When a worker flags a question:
- Write the question to `docs/inbox.json` as `type: "question"` (append to array, auto-increment ID)
- Still set `metadata.question` for in-session visibility
- Commit the inbox.json change so it's visible to the dashboard

**Step 7 (Loop exit — waiting condition):** When the loop exits because all remaining tasks are blocked (inbox items pending — questions or blockers):
- Set waypoint status to `waiting` in the roadmap JSON
- Report to user what's waiting and how to answer (dashboard URL)

### Migration: Blockers → Inbox

The previous `docs/blockers.json` format is retired. Blockers are now inbox items with `type: "blocker"`.

**What changes:**
- Implement skill writes blockers to `docs/inbox.json` instead of `docs/blockers.json`
- Dashboard reads inbox.json for both questions and blockers (single section, type badges distinguish them)
- Dashboard collector no longer reads blockers.json
- Existing blockers.json files (if any) should be migrated to inbox.json format during implementation

**What stays the same:**
- Blockers and questions are still conceptually different — blockers need human action ("create an account"), questions need a text answer ("which approach should I use?")
- The `type` field preserves this distinction for UI and implement skill behavior

### Edge Cases

- **Session crashes while `implementing`**: Status stays stuck. n8n won't trigger (waiting for `waiting`). User notices on dashboard ("active" but nothing happening) and re-invokes manually. Could add staleness heuristic later.
- **Multiple items answered at once**: Handled naturally — Step 3 processes all answered entries in a single loop iteration.
- **User answers in a Claude session instead of dashboard**: Still works — implement's Step 3 checks both metadata.question (in-session) and inbox.json (dashboard). The in-session path clears the inbox entry too.
- **No inbox.json exists**: Skills create it with `[]` on first write. Dashboard returns empty array.
- **Legacy blockers.json exists**: Migration task converts existing entries to inbox.json format. After migration, blockers.json is deleted.
- **Discussion session resolves the item directly**: The discussion skill/command writes the resolution back to inbox.json and sets status to `answered`. Same downstream flow as a dashboard reply.

### What's NOT in Scope

- Push notifications when a question arrives (that's w17)
- Rich formatting in questions or answers (plain text)
- Threaded conversations (one question, one answer per entry)
- Proactive worker messages that don't need a reply (future)
- Full Telegram or remote control integration (evaluated via research spike, implemented if feasible, but not a blocker for w11 completion)
