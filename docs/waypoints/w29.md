# W29: Human Task Inbox Write Guardrail

## Objective

Ensure the implement skill reliably writes human tasks to `docs/inbox.json` by adding a verification check after Step 2 that confirms the write happened.

## Done When

- Step 2 of waypoint-implement includes a self-verification step that reads back `docs/inbox.json` after writing and confirms all unblocked human tasks have corresponding inbox entries
- If verification fails (entries missing), the skill retries the write before proceeding
- Verified working on a real or simulated waypoint with human tasks

## Design

The fix is a small addition to the implement skill's Step 2 in `SKILL.md`.

### Current Step 2 flow:
1. Find tasks with `metadata.assignee == "human"` that are unblocked
2. Present them to the user
3. Write blocker entries to `docs/inbox.json`

### New Step 2 flow (add after the write):
4. **Verify**: Read `docs/inbox.json` back. For each unblocked human task, confirm an entry exists with matching `task_id` and `status: "pending"`. If any are missing, write them now.
5. **Commit**: If any new entries were written (or the file was created), commit `docs/inbox.json` immediately so the dashboard picks it up.

The commit sub-step already exists in Step 6 for worker questions, but Step 2 doesn't explicitly say to commit. Adding an explicit commit instruction reduces the chance of the write being lost.
