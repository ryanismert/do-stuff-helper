# w32: ClaudePluginBuild Integration

## Objective

When a waypoint is fundamentally "build a piece of software," delegate its execution to ClaudePluginBuild's specialized pipeline instead of do-stuff-helper's generic task execution.

## Done When

- `organize` skill ensures ClaudePluginBuild is installed as a project-level plugin when bootstrapping new activities
- `waypoint-design` detects software-build waypoints, recommends delegation to the user, and on confirmation hands off to `/build:prd` with existing context as seed
- `waypoint-implement` recognizes delegated waypoints and either skips them or invokes `/build:implement`
- The handoff is clean: ClaudePluginBuild uses its own task list, and when it finishes, do-stuff-helper marks the waypoint done

## Design

### Detection & Delegation Flow

The intercept point is **waypoint-design, Step 5** (Design the Waypoint). After assessing complexity but before writing the full design doc, the skill evaluates whether the waypoint is primarily a software build:

**Heuristic:** Does the waypoint's objective require writing code as its primary deliverable? Building an app, API, service, CLI tool, UI component — yes. Research, process design, content creation, habit formation — no. Mixed waypoints (e.g., "build and launch the app") should be split: the build part delegates to ClaudePluginBuild, the launch part stays in do-stuff-helper.

When detected, the skill says something like:

> "This waypoint looks like it's primarily a software build. I'd recommend using ClaudePluginBuild to handle the PRD, design, planning, and implementation — it has TDD, multi-lens code review, and CI scaffolding built in. Want me to go that route?"

If the user confirms, the design doc is written as a lightweight delegation doc (see below), and the flow diverges.

### Delegation Design Doc Format

```markdown
# wN: Waypoint Title

## Objective
<one sentence>

## Done When
<acceptance criteria>

## Delegation
This waypoint delegates execution to ClaudePluginBuild.
- **PRD:** docs/prd-<slug>.md
- **Design:** docs/design-<slug>.md
- **Plan:** docs/plan-<slug>.md
- **Implementation:** Managed by /build:implement

## Context for PRD
<extracted brief sections + waypoint-specific requirements that seed the PRD>
```

### PRD Handoff

After saving the delegation design doc, instead of invoking `waypoint-planner` (Step 9), the skill invokes `/build:prd` with the "Context for PRD" section as the `$ARGUMENTS` seed. This gives the PRD skill enough context to skip already-answered questions.

The PRD skill's adaptive behavior (it skips answered questions, accepts "skip") should handle the pre-seeded context. If this proves unreliable in practice, we upgrade to a modified "seeded mode" later.

From there, ClaudePluginBuild's own pipeline takes over: `/build:prd` → `/build:design` → `/build:plan` → `/build:implement`.

### Organize Skill Change

In the organize skill's Step 2 (Install Plugins), add ClaudePluginBuild to the plugin installation list:

1. Ensure the `claude-plugin-build` marketplace is added (if not already)
2. Install the `build` plugin: `claude plugin install build@claude-plugin-build`

### Waypoint-Implement Change

When `waypoint-implement` encounters a waypoint with `delegation: "ClaudePluginBuild"` in the roadmap JSON (or detects a delegation design doc), it should:

1. **If ClaudePluginBuild hasn't started yet:** Invoke `/build:implement` to kick off execution
2. **If ClaudePluginBuild is mid-execution:** Skip the waypoint (it's being handled externally)
3. **If ClaudePluginBuild has finished:** Mark the waypoint as `done`, update changelog, and proceed to the next waypoint

### Roadmap JSON Change

Add an optional `delegation` field to waypoint entries:

```json
{
  "id": "wN",
  "title": "Build the Dashboard App",
  "status": "designed",
  "delegation": "ClaudePluginBuild",
  ...
}
```

This field is set by `waypoint-design` when the user confirms delegation. It signals to `waypoint-planner` (skip — ClaudePluginBuild handles planning) and `waypoint-implement` (delegate to `/build:implement`).

### What Stays Separate

- **Task lists:** ClaudePluginBuild uses its own task list. do-stuff-helper's task list only tracks non-delegated waypoints.
- **Git branches:** ClaudePluginBuild creates `implement/<slug>` branches per its own convention. do-stuff-helper doesn't interfere.
- **Artifacts:** ClaudePluginBuild writes `docs/prd-*.md`, `docs/design-*.md`, `docs/plan-*.md` in the activity repo. These coexist with do-stuff-helper's `docs/waypoints/` files. No name collisions.
