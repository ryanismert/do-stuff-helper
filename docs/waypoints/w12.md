# W12: Forward Motion Analyst

## Objective

Maintain a living week-ahead plan that tells the user what 2-3 things to focus on next, updated automatically as work progresses, with a separate weekly retrospective for reflection.

## Done When

- A plan agent refreshes automatically when progress occurs (waypoint completes, inbox item resolves, roadmap changes) and on the weekly Sunday run
- It produces a **week-ahead plan** with 2-3 priorities, grounded in the user's life goals, current roadmap state, and known tendencies
- The plan starts from life priorities (major goals and life area importance ratings) and maps downward to concrete next steps across activities
- Plan items that persist without progress across multiple refreshes are flagged as stalled — this is the primary staleness signal (not waypoint age)
- A separate **weekly review** runs Sunday, summarizing what moved forward, what didn't, and tendency observations
- The plan is displayed prominently on the dashboard above the inbox and activity cards
- The weekly review is accessible via an expandable panel or link — not front and center
- The agent watches for known tendencies: gravitating to easy work over hard, building software instead of doing uncomfortable tasks, and letting air gaps persist

## Design

### Two Separate Flows

**Flow 1: Plan (on-demand, event-driven)**
- Triggered when: waypoint completes, inbox item resolves, roadmap status changes
- Also triggered by the weekly Sunday run (catches any drift not covered by events)
- Produces/updates `~/exoselfai/docs/forward-motion/current-plan.json`
- Fast — reads state, evaluates priorities, writes plan

**Flow 2: Weekly Review (Sunday only)**
- Runs once per week via n8n (Sunday evening)
- Produces `~/exoselfai/docs/forward-motion/review-YYYY-WNN.md`
- Reflective — looks back at what happened, assesses alignment, notes patterns

### Plan Skill Flow

**Step 1: Gather State**
- Read `~/exoselfai/docs/user-profile.json` — yearly theme, major goals (with life area mappings and success criteria), life area importance ratings, known tendencies
- Scan all `~/exoselfai/activities/*/docs/roadmap-*.json` — waypoint statuses, due dates
- Scan all `~/exoselfai/activities/*/docs/inbox.json` — pending blockers and their age
- Read existing plan if present (`current-plan.json`) — to check what was previously prioritized, when each item was first added, and whether progress occurred

**Step 2: Build the Life Priority Lens**

Start from the user's stated life priorities, not from activities:

1. Read major goals from the profile — each has `life_areas`, `success_criteria`, `timeframe`, `status`
2. Read life area importance ratings (1-7 scale)
3. For each major goal, identify which activities and waypoints advance it
4. Note which major goals have **no active work** advancing them — these are blind spots
5. Note which goals have approaching timeframes (e.g., "lose 10 lbs by June")

This produces a goal-to-activity mapping:
```
Goal: "Launch side businesses" (importance areas: money 7, career 6, adventure 5)
  → wildlife-cards: w3 (implementing), w4 (pending)
  → ai-consulting: no activity exists yet ← blind spot

Goal: "Lose 10 lbs by June" (importance area: health 5)
  → no activity exists ← blind spot, approaching deadline
```

**Step 3: Evaluate Candidates**

Gather all actionable candidates:
- Waypoints ready for design/planning/implementation
- Inbox items waiting on the user
- User-facing tasks (customer outreach, decisions, phone calls)
- Goals with no activity or waypoint advancing them (blind spots to flag)

Score each by:
1. **Goal alignment** — does this advance a major goal? Weight by life area importance
2. **Leverage** — does completing this unblock other work? (dependency graph)
3. **Urgency** — approaching due date or timeframe? Aging blockers?
4. **Tendency correction** — is the user's pattern of avoidance causing this to be skipped? Boost priority if so

**Step 4: Select Top Priorities**

Pick 2-3 items. Rules:
- At most one "comfortable" item (building, coding, designing) — the rest should push toward harder or neglected goals
- Items aligned with the yearly theme ("Year of Launch") get a boost
- If a major goal has no concrete next step, the priority might be "create an activity for X" or "unblock Y by answering inbox item Z"
- If all ready items are genuinely comfortable/easy and harder work isn't actionable, note it but don't manufacture urgency

**Step 4b: Stability Principle**

The plan should feel like a reliable to-do list, not something that reshuffles unpredictably. On each refresh:

- **Preserve the existing priority ranking** unless there's a concrete reason to change it. A new waypoint unblocking or a new activity appearing doesn't automatically bump current priorities — it's noted as available but doesn't reshuffle the list.
- **Reasons to change the list:**
  - A priority item is marked done (rotate in a new one)
  - User gives feedback on an item (defer, skip, blocked — re-evaluate)
  - A due date becomes urgent (approaching deadline displaces a lower-urgency item — analyst explains why)
  - The weekly Sunday run (this is where bigger re-evaluations happen: life priority shifts, new goals, changed circumstances)
- **Reasons NOT to change the list:**
  - A new waypoint became unblocked in some activity
  - A new activity was created
  - Minor progress on a non-priority waypoint
  - Routine inbox items arriving

In practice, mid-week refreshes triggered by completion or feedback are **incremental** — they fill the slot that opened, factoring in all context. The Sunday run is the **full re-evaluation** where the analyst steps back and looks at the whole picture, and may reshuffle more substantially.

**Step 5: Track Plan-Item Staleness**

This is the primary staleness signal — not how long a waypoint has been pending.

For each priority item, track:
- `first_appeared`: date this item first entered the plan
- `refreshes_without_progress`: count of plan refreshes where this item was present and its status didn't change

Thresholds:
- **3+ refreshes without progress**: flag as "lingering" — shown with a visual indicator on the dashboard
- **7+ refreshes**: flag as "stuck" — this is likely a tendency-avoidance pattern or a genuine blocker that needs to be surfaced differently

When an item is flagged as stuck, the tendency analysis in Step 6 should specifically assess whether the user is avoiding it and suggest a concrete unblock action (break it into a smaller first step, ask for help, delegate, or decide to deprioritize it honestly).

**Step 6: Tendency Check**

Read `user-profile.json` → `known_tendencies` and check:

1. **Easy over hard**: Are completed items mostly low-complexity while high-impact priorities linger on the plan?
2. **Building over selling**: For business activities, is progress all on product/code while customer-facing tasks persist as stale plan items?
3. **Air gap persistence**: Inbox items aging beyond 3 days, waypoints in `waiting` with no movement
4. **Self-deprioritization**: Personal goals (fitness, family, social) consistently absent from the plan or always ranked last
5. **Plan-item avoidance**: Items lingering on the plan for 3+ refreshes — the user sees them and doesn't act

Tendency flags are constructive nudges, not guilt trips. Only surfaced when the pattern is actually present. Phrased as: "This has been on the plan for [N] refreshes — [observation about why it might be stuck] — [concrete suggestion to unblock]."

### User Feedback on Plan Items

The user can provide per-item feedback directly from the dashboard. This is the primary way the user steers the plan between refreshes.

**Feedback flow:**
1. User types a short message in the text input beneath a priority item on the dashboard
2. Dashboard POSTs to `POST /api/plan/feedback` with `{ rank, feedback }`
3. Server writes the feedback to the priority item's `user_feedback` field in `current-plan.json`
4. Server triggers a plan refresh — the analyst reads the feedback and re-evaluates priorities

**What the analyst does with feedback:**
- **"Not important right now"** / **"Deprioritize"** → drop the item, replace with the next best candidate. The item won't reappear unless conditions change (e.g., deadline approaches, dependency unblocks something)
- **"Plan to do this in [timeframe]"** / **"Deferred until May"** → set `deferred_until` date on the item. Item is removed from the plan and won't reappear until that date. Shown in a muted "deferred" section on the dashboard so the user remembers it exists
- **"Blocked on [something]"** → analyst notes the blocker. If it's something the system can track (an inbox item, a dependency), it watches for resolution. If it's external, it holds the item with the blocker note visible and checks back on the next weekly run
- **"Already working on this"** / **"Started"** → update status to `in_progress`
- **"Done"** → update status to `done`, trigger refresh to rotate in new priority
- **Free-form context** → stored as `user_feedback`, analyst factors it into the next refresh's reasoning. Examples: "waiting for a call back from the vendor", "need to research options first", "Andrea and I need to discuss this weekend"

**Staleness interaction:** When a user provides feedback, `refreshes_without_progress` resets to 0. The user acknowledged the item — the clock restarts. Staleness only flags items the user is ignoring, not items they've consciously deferred or explained.

**Deferred items on the dashboard:** Items with a `deferred_until` date are shown in a collapsed "Deferred" section below the active priorities — muted, not prominent, but visible so the user remembers what they pushed out.

### Weekly Review Skill Flow

Runs separately on Sundays. Reads the week's plan history and activity changelogs.

Produces `~/exoselfai/docs/forward-motion/review-YYYY-WNN.md`:

```markdown
# Week NN Review — YYYY-MM-DD

## Progress
- [activity]: [what moved forward]

## Stalled
- [activity]: [what didn't move, why if known]

## Effort vs. Priority
[Were the week's priorities addressed? Where did time actually go?]

## Aging Blockers
- [item]: [age], [what's needed]

## Tendency Watch
[Only when a pattern is present — not every week]

## Goal Coverage
[Which major goals got attention, which didn't, blind spots identified]
```

### Trigger Mechanisms

**Incremental refresh (on-demand, event-driven):**
- When `waypoint-implement` marks a waypoint `done` → fill the open slot
- When an inbox item is resolved via dashboard → update if the resolved item was a priority
- When the user submits feedback on a plan item → re-evaluate that item (defer, drop, or keep with context)
- These are lightweight and follow the stability principle: preserve rankings, only change items with a concrete reason

**Full re-evaluation (weekly, n8n):**
- Schedule: Sunday evening (e.g., 8:00 PM)
- POST to webhook server → invoke forward-motion skill in review mode (produces weekly review, then does a full plan re-evaluation for the new week)
- This is where bigger reshuffles happen: life priority shifts, new activities, changed circumstances, accumulated context from the week's feedback

No daily cron needed — the event-driven triggers keep the plan current, and the Sunday run catches any drift.

### Dashboard Changes

**"This Week" panel** — displayed above the inbox and activity cards:
- Shows the 2-3 priority items from `current-plan.json`
- Each priority: summary, activity tag, rationale, status badge (not started / in progress / done)
- **Per-item feedback input**: a text field beneath each priority item. User types a message and hits enter. The message is sent to the API, stored as `user_feedback`, and triggers a plan refresh. Placeholder text: "Not now, blocked, done, or any context..."
- Completed priorities shown with strikethrough or muted style
- Lingering items (3+ refreshes) shown with a subtle indicator
- Stuck items (7+ refreshes) shown with a stronger indicator and the tendency flag
- Aging user blockers shown below priorities with age badges (yellow > 3 days, red > 7 days)
- Blind spots shown as a subtle note: "No active work toward [goal]"
- **Deferred section**: collapsed by default, shows items the user pushed out with their `deferred_until` date. Muted styling, not prominent

**Weekly review** — expandable panel or link below the plan:
- "Last week's review" toggle that expands inline
- Not prominent — there for reflection when the user wants it

### Where Artifacts Live

```
~/exoselfai/docs/forward-motion/
├── current-plan.json          # Rolling plan (refreshed on-demand + weekly)
└── review-YYYY-WNN.md         # Weekly reviews (one per week, archive)
```

Lives in the main exoselfai repo because it spans all activities.

### What's NOT in Scope

- Chat/notification delivery (w17)
- Auto-resolving blockers (w14)
- Reprioritizing activities or changing roadmaps (advisory only — surfaces observations)
- Calendar integration
- Email integration
