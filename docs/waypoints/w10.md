# W10: Dashboard — Core

## Objective

Provide a single view of all activities — status, recent changes, blockers, next actions, and active work — so the user can stay oriented without diving into individual repos.

## Done When

- A dashboard page shows all activities with:
  - Current phase and waypoint progress (counts and percentages)
  - Recent changelog (waypoint-level milestones: designed, planned, completed)
  - Blockers assigned to the user (human tasks)
  - Next actions (unblocked waypoints ready for design or execution)
  - Whether a waypoint is actively being implemented
- Dashboard auto-refreshes and is accessible from any device on Tailscale
- Data is no more than 5 minutes stale
- All lifecycle skills (discover, design, plan, implement, replan, roadmap) write milestone entries to the changelog

## Design

### Architecture

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Data Collector  │────▶│  Dashboard API    │────▶│  Dashboard UI   │
│  (reads repos)   │     │  (Express routes) │     │  (Static HTML)  │
└─────────────────┘     └──────────────────┘     └─────────────────┘
```

Runs as a Docker service on port 3002, accessible via Tailscale at `exoself-dev1:3002`.

**Data Collector** scans `~/exoselfai/activities/*/` and for each activity:
- Reads `docs/roadmap-<slug>.json` for waypoint statuses, phases, dependencies
- Reads `docs/changelog.md` for recent milestone entries (filtered by pattern)
- Reads `docs/blockers.json` for human-assigned tasks
- Computes next actions: pending waypoints with all deps done
- Detects active work: any waypoint with status `implementing`

**Dashboard API** — Express routes:
- `GET /api/dashboard` — full aggregated data as JSON
- `GET /` — serves the dashboard HTML page

**Dashboard UI** — Single static HTML page with vanilla JS/CSS:
- Polls `/api/dashboard` every 60 seconds
- Activity cards with progress, changelog, blockers, next actions
- "Working now" indicator when any waypoint is `implementing`
- Responsive layout (usable on phone)

### Waypoint Statuses

Replace the loosely-used `in-progress` with precise statuses:

| Status | Meaning |
|--------|---------|
| `pending` | Not started |
| `implementing` | A worker/agent is actively executing this |
| `done` | Completed |
| `obsolete` | No longer relevant |

The `waypoint-implement` skill sets `implementing` when it starts work and `done` when it finishes. The dashboard derives "activity is active" from any waypoint being in `implementing` status.

### Changelog Convention

All lifecycle skills write milestone entries to `docs/changelog.md` using a consistent format:

```
## YYYY-MM-DD — <Verb> <scope>
<1-2 sentence human-readable summary>
```

Entries can be **activity-level** (no waypoint ID) or **waypoint-level**:

```markdown
## 2026-02-25 — Activity defined: Do-Stuff-Helper
Expert interview completed, brief saved.

## 2026-02-25 — Designed w10: Dashboard — Core
Single-page dashboard showing activity status, changelog, blockers, and active work.

## 2026-02-25 — Completed w9: Backlog & Replanning Process
Created replan skill and backlog conventions for capturing out-of-scope work.
```

**Verbs by skill:**

| Skill | Verb | Scope |
|-------|------|-------|
| `discover` | **Activity defined** | activity name |
| `roadmap` | **Roadmap created** | activity name |
| `waypoint-design` | **Designed** | waypoint ID + title |
| `waypoint-planner` | **Planned** | waypoint ID + title |
| `waypoint-implement` | **Implementing** (start) / **Completed** (end) | waypoint ID + title |
| `replan` | **Replanned** | summary of changes |

New entries are prepended (newest first). The dashboard filters to lines matching `## YYYY-MM-DD —` to extract the milestone timeline. Existing task-level detail from workers (written below milestone entries) is preserved for history but not shown on the dashboard.

### Smart waypoint-implement

Update `waypoint-implement` to assess work complexity and choose its approach:

- **Complex waypoint** (multiple parallelizable tasks) — full worker pipeline with worktrees, as today
- **Simple waypoint** (a few straightforward changes) — executes directly in the current session, no worker spawning

Either path handles the full lifecycle: sets `implementing`, does the work, writes changelog, sets `done`. This eliminates the temptation to skip the skill for simple waypoints.

### Blocker Convention

**`docs/blockers.json`** per activity (temporary bridge until w11 Inbox):

```json
[
  {
    "id": "blocker-1",
    "waypoint": "w6",
    "title": "Choose pilot activity",
    "description": "User needs to decide between wildlife cards and personal branding",
    "created": "2026-02-25",
    "status": "open"
  }
]
```

Workers write here for human-required tasks. Dashboard reads it. Users resolve by updating status or the implement skill clears them when work resumes.

### Docker Service

```yaml
exoself-dashboard:
  image: node:20-alpine
  volumes:
    - ./scripts/dashboard:/app
    - ./activities:/activities:ro
  ports:
    - "3002:3002"
  working_dir: /app
  command: node server.js
  restart: unless-stopped
```

### Skill Updates Required

These skills need changelog-writing steps added:

1. **discover** — After saving the brief, prepend "Activity defined" entry
2. **roadmap** — After saving the roadmap, prepend "Roadmap created" entry
3. **waypoint-design** — After Step 8 (save), prepend "Designed" entry
4. **waypoint-planner** — After Step 5 (create tasks), prepend "Planned" entry
5. **waypoint-implement** — Prepend "Implementing" at start, "Completed" at end. Add complexity assessment to choose inline vs worker execution.
6. **replan** — After Step 5 (apply changes), prepend "Replanned" entry

### Data Sources

| Data | Source | How |
|------|--------|-----|
| Waypoint status | `docs/roadmap-<slug>.json` | Read JSON, compute stats |
| Recent activity | `docs/changelog.md` | Parse milestone entries |
| Blockers | `docs/blockers.json` | Read JSON |
| Next actions | Roadmap JSON | Filter: status=pending, all deps done |
| Active work | Roadmap JSON | Any waypoint with status=implementing |

### What's NOT in Scope

- SSH entry points into active sessions (added to backlog)
- Drill-down into individual waypoints or task lists
- Editing or resolving blockers from the dashboard (read-only for MVP)
- Metrics/trends over time (Grafana + Prometheus handle that)
- Inbox/messaging (w11)

### File Structure

All dashboard and inbox code lives in the **main exoselfai repo** (not do-stuff-helper). If this ever needs to be extracted, here's what to move:

```
~/exoselfai/
├── scripts/
│   ├── dashboard/                   # Dashboard service (Docker container)
│   │   ├── server.js                # Express server + inbox API endpoints
│   │   ├── lib/
│   │   │   └── collector.js         # Activity scanning and data aggregation
│   │   ├── public/
│   │   │   └── index.html           # Dashboard UI (single page, vanilla JS/CSS)
│   │   └── package.json
│   ├── discuss.js                   # Tmux-based Claude remote-control session manager
│   ├── webhook-server.js            # Webhook API server (systemd service, port 3001)
│   │                                  Hosts discuss session endpoints + task execution
│   └── logger.js                    # Structured logging (shared by webhook + dashboard)
├── docker-compose.yml               # exoself-dashboard service definition (port 3002)
└── config/systemd/
    └── claude-webhook.service       # Systemd unit for webhook-server.js
```

**Why it lives there:** The dashboard reads from all activities and the webhook server is a host-level service — both are infrastructure, not specific to any single activity.
